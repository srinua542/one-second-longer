# Arcade Physics is cosmetic-only — no gameplay-authoritative feedback

Phaser Arcade Physics bodies are used exclusively for cosmetic effects on the client (debris scattering, particle knockback, camera-shake triggers). No Arcade Physics body ever influences a value that enters `GameState` — no position, velocity, or collision result from an Arcade body feeds into scoring, death detection, entity movement, or any other gameplay-authoritative outcome.

All gameplay-authoritative physics (player movement, gravity, platform collision, hazard/collectible overlap) is custom deterministic AABB math in `simulation/`, which runs identically on the client and the headless server. Arcade Physics literally cannot run in Node (it depends on Phaser's scene lifecycle), and replay verification (ADR-0006) requires bit-identical simulation output across both environments. These two constraints together make Arcade Physics structurally impossible to use for any gameplay-affecting computation.

Cosmetic Arcade Physics objects are not simulation entities — they are not in `GameState.entities`, not stepped by `simulation.step()`, not included in replay hashes, and not reported back to the simulation. They live entirely in `client/`, spawned by reacting to `SimEvent` emissions (e.g. a `'death'` event triggers a particle burst). Their visual variation is driven by the cosmetic PRNG stream, which is also excluded from `GameState` and replay hashing (ADR-0008).

Considered alternative: using Arcade Physics for gameplay collision alongside custom physics for the deterministic subset. Rejected because maintaining two physics systems with different authority levels (some collisions "real," others "cosmetic") is a classification problem that's easy to get wrong silently — a single miscategorized collision would cause replay verification failures or leaderboard score corruption with no obvious diagnostic trail.

Consequence: the pause-freeze guarantee for cosmetic physics depends on Phaser's scene-scoped event wiring — Arcade's `World.update()` is invoked by `GameScene`'s own `update`/`postupdate` events, so `GameScene.scene.pause()` silences those events and physics stops as a side effect. This is inferred from Phaser 4's documented architecture, not from an explicit API contract. It must be verified empirically against the pinned Phaser version (4.2.0+) and re-verified on any future version bump. Tweens scoped to `GameScene` should also freeze under scene pause, but this is a separate subsystem with its own event wiring and must be tested independently — especially to confirm that `UIScene`'s tweens remain active while `GameScene` is paused.
