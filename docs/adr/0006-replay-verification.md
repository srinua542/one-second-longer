# Replay verification over score-trusting for leaderboards

Daily Bin leaderboard scores are verified by server-side re-simulation, not by trusting client-reported results. The client submits `{ playerId, seedDate, inputLog }` on session end. The server runs the shared simulation core (ADR-0004) headlessly against that seed and input log, computes the authoritative furthest-%, deaths, and bits banked, and ranks that computed result. The client's claimed score is used only for optimistic UI before server confirmation.

Considered alternatives: trusting client scores (zero backend cost, but any browser devtools user can submit fake scores — worthless for a speedrun-adjacent daily leaderboard); heuristic anti-cheat (speed caps, statistical anomaly detection — adds complexity without guarantees, and false positives erode player trust); portal-native leaderboards (Poki/CrazyGames SDK leaderboard APIs — no verification capability, just score storage).

Replay verification is possible *only because* the game is fully deterministic by design: seeded PRNG with pre-rolled outcomes (ADR-0005), fixed-step physics, and a pure simulation core with no rendering dependencies (ADR-0004). These architectural decisions were made partly to enable this one — determinism is not just a gameplay feature, it's the anti-cheat mechanism.

The trade-off: a backend (Fastify + PostgreSQL + Redis) must be built and maintained. Headless re-simulation of a 5-minute session runs in a fraction of a second, so compute cost per submission is trivial. The real ongoing cost is operational maintenance time, not infrastructure spend.

Consequence: client submissions include `simVersion` (the simulation code version that produced the run). The server checks this against its current `SIM_VERSION` before attempting replay. A mismatch — e.g. a player finishes a run moments before a deploy and submits after — results in an explicit rejection (`'version-mismatch'`), not a silent score-mismatch false-reject. "Deterministic" guarantees same-input-same-output *for a fixed version of the code*; it says nothing across versions. This is the same deploy-boundary problem Factorio, Fortnite replays, and seeded-speedrun communities all solve with version-stamping. Multi-version replay support (keeping old `step()` logic alive) is deferred to v1; reject-and-log is sufficient until analytics show meaningful submission volume landing in the deploy-gap window.
